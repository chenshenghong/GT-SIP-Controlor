# 組播多監聽區（Multi-zone）功能需求單（GT-SIP-GW）

> 提交對象：設備/韌體原廠
> 背景：校園場域的廣播群組天然重疊——同一顆喇叭同時屬於「全校」「年級」「大樓」「單班」等多個廣播範圍。現行韌體組播接收為**單一監聽位址槽**（`/set/sip/multicast` 一組 address:port），喇叭一次只能聽一個組播域，重疊群組無法同時成立；平台側被迫把組播域做成「喇叭分割」、其餘層級退回單播。業界同類 SIP 組播終端（多廠牌 paging speaker/adapter）標配 10–50 組多監聽區（multicast zones），各區帶優先權。
> 依據：現行 REST `/set/sip/multicast` 契約與 `request_set_sip_multicast()` 實作（見 `GT-SIP-REST_API.md`、`docs/firmware-reference/websetsip.utf8.c`）＋ v2.1.1 韌體真機行為。
> 提出日期：2026-07-18（本版：定稿，可直接開工）
> 優先序：🔴 P0 本版次核心（新韌體版次；不阻擋現行出貨——現行單槽模式仍為向下相容基準）

## 一、總述

**目標**：把組播接收從「單一位址槽」擴充為一張**16 區監聽區表（multicast zones）**——每區獨立的 `multicast_address:port`、**優先權**、啟用開關、音頻編碼；裝置同時監聽全部啟用區（每區各自 IGMP join），**依優先權單流播放、不混音**。config 存放與開機恢復完整鏡像現行組播設定模式（`/etc/ifcfg-sip`、`sip.sdk` 無參數重讀通知），把原廠改動與風險降到最低。

**規格底線（定案，非討論項）**：

- **區數＝16**（固定值；若貴廠因記憶體/執行緒預算無法達到，請於回覆回填實際可支援上限與依據——見 §十，僅此一項需回填）。
- 每區欄位：
  | 欄位 | 型別／選項 | 說明 |
  |---|---|---|
  | `zone_id` | 1–16 | 邏輯區編號，固定 |
  | `multicast_address` | 字串 | `224.0.0.0`–`239.255.255.255` |
  | `multicast_port` | 整數 | 建議 1024–65535 |
  | `priority` | 整數 1–16 | 數字越小優先權越高；**16 區全域唯一**（僅計入 `enabled:true` 的區；停用區的 priority 值不參與唯一性檢查，允許沿用/待命） |
  | `enabled` | boolean | 啟用開關 |
  | `audio_codec` | `G.711U` / `G.722` | 每區獨立可選（不做全域單一 codec，實作成本差異可忽略） |
- **搶佔語意**：高優先權區來流 →**即時搶佔**（≤200ms）停止低優先權播放、切至高優先權流；高優先權流結束（**RTP 靜默逾時，預設 2 秒、可配置**）後，若低優先權流仍在送，**恢復播放**低優先權流。
- **不混音**：任一時刻僅單流出聲，切流瞬間不得殘留前一流取樣（避免疊音/爆音）。
- **同優先權先到先播**：同優先權兩流同時存在，先到者播、後到者不搶佔；此行為固定，非可配置項。

**對映校園場景**：zone 1＝緊急（最高優先權）、zone 2＝全校鐘聲、zone 3＝年級、zone 4＝大樓……緊急蓋掉鐘聲的行為即由裝置端優先權原生保證，平台側不需額外仲裁。

**責任切分（沿用現行三層，鏡像 IO 觸點需求單模式）**：

```
瀏覽器 device-web「📡 組播監聽區」頁 ──（我方已完成，隨交付包提供）
   │ GET /get/sip/multicast/zones、POST /set/sip/multicast/zones（Bearer token）
   ▼
websetsip.c（控制面）──（原廠：新增 2 條路由，照既有 request_get_sip_config /
   │                        request_set_sip_multicast 模板）
   │ 讀寫 /etc/ifcfg-sip 的 multicast_zones（JSON 字串，見 §三）
   │ Unix socket /tmp/sip.sdk：{"command":"set_multicast_zones","cseq":n}\r\n\r\n
   ▼
/opt/termapp（執行面）──（原廠：主要工作量）
   多 socket join 執行緒（每啟用區各自 IGMP membership）
     → 播放仲裁器（單一輸出路徑，依優先權切流／搶佔／逾時恢復）→ 音頻輸出
```

## 二、向下相容（驗收硬條款，本次不動）

1. 現行 `POST /set/sip/multicast`（單槽 payload：`multicast_address`/`multicast_port`/`enabled`/`audio_codec`）**必須繼續有效**，語意＝寫入 **zone 1**（其餘區不受影響）。既有平台 provisioning 整合零改動即可運作。
2. `GET /get/sip/multicast` 維持現行回傳形狀（`multicast_config` 物件，回 zone 1 的值），與新多區端點並存。
3. **舊韌體 config 沒有 `multicast_zones` 這個 key 時**＝視為只有 zone 1（沿用舊有 `MULTICAST_ADDRESS`/`MULTICAST_PORT`/`MULTICAST_ENABLED`/`MULTICAST_CODEC` 四個既有 key），裝置行為與現行單槽韌體**完全一致**，不得因新增此功能而在既有欄位缺 `multicast_zones` 時退化或報錯。
4. device-web「SIP / 組播」頁既有「組播接收」卡片維持原樣（見 device-web §變更），標註為「＝zone 1 快速設定」，兩個編輯入口（單槽卡片／多區頁）互相反映同一份 zone 1 資料。

## 三、Config Schema

### 3.1 儲存位置與格式

`/etc/ifcfg-sip` 既有 key-value 檔案新增一個 key **`MULTICAST_ZONES`**，值為 **JSON 陣列字串**（不逐欄位拆成 key-value——16 區 × 5 欄位若逐欄位化將產生 80 個 key，不利維護／擴充，且與現行 `key_value_file` 純量值慣例不合；比照 IO 觸點需求單 `io_config` 的做法，整表存一個 JSON 字串 key，讀寫時用 cJSON 序列化/反序列化）。

```json
[
  {"zone_id":1,"multicast_address":"239.168.12.1","multicast_port":2000,"priority":1,"enabled":true,"audio_codec":"G.722"},
  {"zone_id":2,"multicast_address":"239.168.12.9","multicast_port":2000,"priority":2,"enabled":true,"audio_codec":"G.722"},
  {"zone_id":3,"multicast_address":"239.168.12.17","multicast_port":2000,"priority":3,"enabled":false,"audio_codec":"G.711U"}
  /* … 補滿至 zone_id 16，未配置區 enabled:false、位址可留空 */
]
```

- `MULTICAST_ZONES` 缺失（舊韌體/尚未寫入）＝依 §二第 3 點，退回僅 zone 1（讀舊有四個 key）。
- 一旦 `MULTICAST_ZONES` 存在，**zone 1 的讀寫以此陣列內 `zone_id:1` 的項目為準**、舊四個 key 停止作為讀取來源，但 `POST /set/sip/multicast`（單槽端點）仍寫入舊四個 key**並同步覆寫** `MULTICAST_ZONES` 陣列中 `zone_id:1` 的對應欄位（保持兩份資料一致，見 §七）。

### 3.2 欄位定義

同 §一欄位表；`zone_id` 為陣列索引依據，非可變更（1–16 固定）。

## 四、REST API（原廠新增 2 條，照 `request_get_sip_config` / `request_set_sip_multicast` 模板）

沿用既有 `:80`/HTTPS 管理服務與 Bearer token 鑑權（GET 亦須驗 token，比照 SEC-01 要求）。GET 回應**不含** `status` 外層（比照 `request_get_sip_config` 現行行為，直接回資料物件）；POST 回應沿用既有 `{"status":"success"/"error", "message"/"details", ...}` 慣例。

### 4.1 `GET /get/sip/multicast/zones`

```json
{ "zones": [
    { "zone_id": 1, "multicast_address": "239.168.12.1", "multicast_port": 2000,
      "priority": 1, "enabled": true, "audio_codec": "G.722" },
    { "zone_id": 2, "multicast_address": "239.168.12.9", "multicast_port": 2000,
      "priority": 2, "enabled": true, "audio_codec": "G.722" }
    /* … 共 16 筆，MULTICAST_ZONES 缺失時回傳僅 1 筆（由舊四個 key 合成 zone_id:1） */
] }
```

回應須為**合法 JSON**（cJSON 建構、欄位值 escape，比照 SEC-09）。

### 4.2 `POST /set/sip/multicast/zones`

Payload：`{ "zones": [ {zone_id,multicast_address,multicast_port,priority,enabled,audio_codec} × 16 ] }`。**整表覆寫**（陣列同上形狀，缺的 `zone_id` 視為該區維持 `enabled:false`／原值不變，是否採「整表覆寫」或「缺即維持原值」由原廠依實作成本擇一，但**須與 device-web 行為一致並於回覆說明採用哪一種**——device-web 一律送完整 16 筆，兩種語意在此情形下等價）。

**伺服器端驗證（拒絕整批寫入、回 `status:"error"` + `error_code:"E001"`，錯誤訊息需指出是哪一筆 `zone_id`，比照現行 `request_set_sip_multicast` 的 E001 慣例）**：

- `multicast_address` 格式合法且首位元組落在 224–239（同現行 `request_set_sip_multicast` 的 `ip_first` 檢查）。
- `multicast_port` 落在 1024–65535。
- `priority` 為 1–16 整數；**所有 `enabled:true` 的區之間 `priority` 不得重複**（此為伺服器端最終防線，device-web 前端已做即時檢查，但不可只靠前端）。
- `audio_codec` ∈ {`G.711U`,`G.722`}。
- `enabled:false` 的區：若該區為**全空佔位列**（`multicast_address` 為空字串、`multicast_port`/`priority` 皆為 0、`audio_codec` 為空）＝尚未配置，**略過格式檢查**、原樣存入；若該區帶有任何非空值（曾配置過、現僅暫時停用），則比照 `enabled:true` 走全部格式檢查，**不可讓半成品垃圾值殘留、待日後重新啟用時才炸開**。device-web 端已依此規則實作（見 device-web §變更）。

驗證通過後：寫入 `MULTICAST_ZONES`；若 `zone_id:1` 有變動，**同步覆寫**舊四個 key（`MULTICAST_ADDRESS`/`MULTICAST_PORT`/`MULTICAST_ENABLED`/`MULTICAST_CODEC`，見 §三 3.1）；透過 `/tmp/sip.sdk` 送通知（見 §五）立即生效免重啟。

> device-web 端「📡 組播監聽區」頁**已由我方完成並隨交付包提供**，直接對接上述兩支端點；韌體未支援時頁面容錯顯示「此韌體未回報多監聽區設定」，原廠實作後即自動生效。

## 五、`sip.sdk` 通知

寫入 `MULTICAST_ZONES` 成功後，經 Unix socket `/tmp/sip.sdk` 送：

```
{"command":"set_multicast_zones","cseq":n}\r\n\r\n
```

`/opt/termapp` 收到後**無參數重讀** `/etc/ifcfg-sip` 的 `MULTICAST_ZONES`，重建 join 集合與播放仲裁器狀態，**不得重啟程序或中斷現有 SIP 通話**（同現行 `set_sip_*` 系列通知的行為基準）。

## 六、韌體實作要求（termapp / kernel）

| 編號 | 要求 |
|---|---|
| **MZ-01** | 多 socket 同時 join 各 `enabled:true` 區的 multicast group（每區獨立 IGMP membership，`setsockopt(IP_ADD_MEMBERSHIP)`），一區 join 失敗不得影響其餘區。 |
| **MZ-02** | 播放仲裁器：單一輸出路徑，依 §一搶佔語意切流；切流時**清空前一流的 jitter buffer / 取樣殘留**，避免疊音/爆音。搶佔延遲 ≤200ms（自高優先權首個 RTP 封包抵達起算）。 |
| **MZ-03** | RTP 靜默逾時判定：一區持續 `RTP_SILENCE_TIMEOUT_MS`（預設 2000ms，建議可配置、範圍 500–10000ms）未收到新 RTP 封包＝該流結束，觸發低優先權流恢復判斷（若仍在送）。 |
| **MZ-04** | RTP 時鐘處理沿用 v2.1.1 修正後行為（G.722 timestamp 8kHz 時鐘，RFC 3551 §4.5.2）——多區不得回歸此 bug；16 區任一區用 G.722 皆須正確。 |
| **MZ-05** | 開機自動恢復：讀取 `MULTICAST_ZONES`，對全部 `enabled:true` 區重新 join；若 `MULTICAST_ZONES` 不存在，回退只 join 舊 zone 1（同現行單槽行為）。 |
| **MZ-06** | 資源約束下（CPU/記憶體/socket 上限）優先保證：**已啟用區的 join 完整性 ＞ 區數上限**——即若貴廠評估無法穩定支援 16 區同時 join，請如實回填實際上限（§十），而非讓超額區靜默失敗。 |
| **MZ-07** | 效能護欄同 IO 需求單 IO-04：播放仲裁器/join 管理全部非同步，**禁止在音頻回呼裡做網路 I/O 之外的阻塞操作**（單核嵌入式 CPU，音訊即時性關鍵）。 |

## 七、與現有介面的一致性（重要）

1. zone 1 的資料經三條路徑對外（舊單槽端點 `/get|set/sip/multicast`、新多區端點 `zones[0]`、DBP 能力清單）時，**必須來自同一份執行期資料源**，任一路徑修改後其他路徑 ≤2 秒內一致、不得要求重啟。此要求與《SIP 資訊一致性需求單》同源，請一併落實。
2. 16 區配置與各區即時收流狀態（是否正在播放）建議進 DBP 能力清單與 REST 回報，便於平台側監控與日誌（可列為 P1，不阻擋本版次驗收）。

## 十一、與平台側的整合（供參考，我方自理）

平台（OmniVox）資料模型已允許同一喇叭隸屬多個組播域（`sip_multicast_group_members` 複合鍵），provisioning 服務將於韌體支援後擴充為多區寫入；緊急/鐘聲優先權將直接映射 zone priority。本需求落地後，重疊群組全層級皆可組播嚴格同步，無需單播補層。
