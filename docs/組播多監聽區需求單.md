# 組播多監聽區（Multi-zone）功能需求單（GT-SIP-GW）

> 提交對象：設備/韌體原廠
> 背景：校園場域的廣播群組天然重疊——同一顆喇叭同時屬於「全校」「年級」「大樓」「單班」等多個廣播範圍。現行韌體組播接收為**單一監聽位址槽**（`/set/sip/multicast` 一組 address:port），喇叭一次只能聽一個組播域，重疊群組無法同時成立；平台側被迫把組播域做成「喇叭分割」、其餘層級退回單播。業界同類 SIP 組播終端（多廠牌 paging speaker/adapter）標配 **10–50 組多監聽區（multicast zones），各區帶優先權**。
> 依據：現行 REST `/set/sip/multicast` 契約（見 `GT-SIP-REST_API.md`）＋ v2.1.1 韌體真機行為。
> 提出日期：2026-07-18
> 優先序：🟠 P1 應做（新韌體版次；不阻擋現行出貨——現行單槽模式仍為向下相容基準）

---

## 一、總述

**目標**：把組播接收從「單一位址槽」擴充為一張**多監聽區表（zones）**——每區獨立的 `multicast_address:port`、**優先權**、啟用開關；裝置同時監聽全部啟用區，**依優先權單流播放、不混音**。config 存放與開機恢復完整鏡像現行組播設定模式（`/etc/ifcfg-sip` 或既有 config 分區、`sip.sdk` 無參數重讀通知），把原廠改動與風險降到最低。

**規格底線**：

- **區數 ≥ 8**（建議 16；請原廠依記憶體/執行緒預算回覆實際上限）。
- 每區欄位：`zone_id`（1..N）、`multicast_address`、`multicast_port`、`priority`（數字越小越高，建議 1..N 全域唯一）、`enabled`、`audio_codec`（沿用現行合法值 `G.711U`/`G.722`；若全域單一 codec 較易實作，請於回覆說明，我方可接受全區共用 codec）。
- **搶佔語意**：高優先權區來流 → **即時**（≤200ms）停止低優先權播放、切至高優先權流；高優先權流結束（RTP 靜默逾時，建議 2s 可配）後，若低優先權流仍在送，**恢復播放**低優先權流。
- **不混音**：任一時刻僅單流出聲。
- 同優先權同時來流：後到不搶佔（維持先到），行為需固定並文件化。

**對映校園場景**：zone 1＝緊急（最高優先權）、zone 2＝全校鐘聲、zone 3＝年級、zone 4＝大樓……緊急蓋掉鐘聲的行為即由裝置端優先權原生保證。

## 二、向下相容（重要，驗收硬條款）

1. 現行 `POST /set/sip/multicast`（單槽 payload：`multicast_address`/`multicast_port`/`enabled`/`audio_codec`）**必須繼續有效**，語意＝寫入 **zone 1**（其餘區不受影響）。既有平台 provisioning 整合零改動即可運作。
2. `GET /get/sip/multicast` 維持現行回傳形狀（回 zone 1），另新增多區端點（見 §三）。
3. 未配置多區（只有 zone 1）時，裝置行為與現行單槽韌體**完全一致**。

## 三、REST API（照既有 `websetsip.c` request_* 模板新增）

- `GET /get/sip/multicast/zones`（Bearer token）→
  ```json
  { "status": "success", "data": { "zones": [
      { "zone_id": 1, "multicast_address": "239.168.12.1", "multicast_port": 2000,
        "priority": 2, "enabled": true, "audio_codec": "G.722" },
      { "zone_id": 2, "multicast_address": "239.168.12.9", "multicast_port": 2000,
        "priority": 1, "enabled": true, "audio_codec": "G.722" }
  ] } }
  ```
- `POST /set/sip/multicast/zones`（Bearer token）：整表覆寫（陣列同上形狀）；寫入即生效免重啟（`sip.sdk` 重讀通知，同現行模式）；欄位驗證失敗回 `status:"error"` ＋ `error_code`（沿用既有錯誤碼慣例）。
- 回應一律真實反映寫入結果（HTTP 200 但 `status` 非 success 視為失敗——同現行契約注意事項）。

## 四、韌體實作要求

1. 多 socket 同時 join 各啟用區的 multicast group（IGMP membership 每區獨立）。
2. 播放仲裁器：單一輸出路徑，依 §一搶佔語意切流；切流時**不得殘留前一流的取樣**（避免疊音/爆音）。
3. RTP 時鐘處理沿用 v2.1.1 修正後行為（G.722 timestamp 8kHz 時鐘，RFC 3551 §4.5.2）——多區不得回歸此 bug。
4. 開機自動恢復全部區設定並重新 join。
5. 資源約束下（如 CPU/記憶體不足以 16 區）優先保證：已啟用區的 join 完整性 > 區數上限。

## 五、端對端驗收清單

| # | 場景 | 預期 |
|---|---|---|
| 1 | 舊版單槽 payload 寫入 | zone 1 生效、行為與現行韌體一致 |
| 2 | 配置 2 區（優先權 2/1），先送低優先權流 | 播低優先權 |
| 3 | 低優先權播放中，高優先權來流 | ≤200ms 切至高優先權、無疊音 |
| 4 | 高優先權流結束、低優先權仍在送 | 於靜默逾時內恢復低優先權播放 |
| 5 | 同優先權兩流同時 | 先到者播、後到不搶佔 |
| 6 | 全部 N 區啟用、輪流送流 | 每區皆可收播（IGMP join 完整） |
| 7 | 斷電重啟 | 全區設定與 join 自動恢復 |
| 8 | G.722 於任一區播放 | 音調/語速正常（無 v2.1.0 2× 回歸） |

## 六、請原廠回覆的問題

## 七、與平台側的整合（供參考，我方自理）

平台（OmniVox）資料模型已允許同一喇叭隸屬多個組播域（`sip_multicast_group_members` 複合鍵），provisioning 服務將於韌體支援後擴充為多區寫入；緊急/鐘聲優先權將直接映射 zone priority。本需求落地後，重疊群組全層級皆可組播嚴格同步，無需單播補層。
