## 一、總述

**目標**：設備透過自身 **Mic 輸入採集音訊 → 編碼 → RTP → UDP 推流到組播位址**；其他 GT-SIP-GW 設備用**既有的組播接收功能**設到同一位址即可收聽。等同把任一台設備變成廣播喊話源。

**架構與責任切分**（沿用現行三層）：

```
瀏覽器 device-web「組播發送」卡片 ──（我方已完成）
   │ POST /set/multicast/tx（Bearer token）
   ▼
websetsip.c（控制面）──（原廠：新增 1 條路由 + 2 處 GET 擴充，照既有模板）
   │ 寫 /etc/ifcfg-sip（MULTICAST_TX_*）
   │ Unix socket /tmp/sip.sdk：{"command": "set_sip_multicast_tx","cseq": 1}\r\n\r\n
   ▼
/opt/termapp（執行面）──（原廠：主要工作量）
   重讀 config → 啟停 TX 管線：Mic 採集（既有，CAP_VOL 生效）
     → 編碼器（既有 OPUS/G.722/G.711）→ RTP 封包化（既有）→ UDP sendto 組播位址（新）
```

**設計原則：完整鏡像既有「組播接收」**——config keys、API 形狀、sip.sdk 無參數重讀通知、開機自動恢復，全部與 `MULTICAST_*`／`/set/sip/multicast` 對稱，將原廠改動與風險降到最低。termapp 為 SIP 通話已具備「Mic→編碼→RTP」全鏈路，本功能只是把出口改接組播 UDP socket。

#### §2.1　`POST /set/multicast/tx`（MTX-02 規格）

- **鑑權**：`Authorization: Bearer <token>`，照既有 9 處內聯驗證模板（含 `strlen(now_token)-1` 之 `-1`）；無/錯 token 回 `A003`、過期回 `A002`。
- **Request**（4 欄全必填，與 `/set/sip/multicast` 對稱）：
  ```json
  {"multicast_address": "239.0.0.100", "multicast_port": 9000,
   "enabled": true, "audio_codec": "G.722"}
  ```
- **驗證**：位址第一段 224–239（否則 `E001`「非法组播地址」）；port 1–65534（否則 `E001`「非法组播端口」）；缺欄/型別錯 `E001`；`audio_codec` **v1 白名單僅 `G.722`**（否則 `E001`；欄位保留是為與 RX API 對稱及未來擴充）。
- **落盤與生效**：照 save_flag 模式——值有變才 `write_keyvalue_file` 並送 sip.sdk 指令 `{"command": "set_sip_multicast_tx","cseq": 1}\r\n\r\n`（無參數，termapp 重讀 `MULTICAST_TX_*`），**熱生效、不重啟**。
- **Response**：既有模板 `{"status": "success","message": "操作成功","data": {}}`；HTTP 一律 200、`charset=GBK`。
- ⚠ **路由整合三注意**：① `request_url` 陣列 `[19]`→`[20]`、新增 `request_url[19]="/set/multicast/tx"`、`init_web_listen(..., request_url, 20, ...)` 三處同步；② dispatch 為 `strncmp(常量, url, len)` 前綴比對——本路徑刻意取 `/set/multicast/tx`（與既有 19 條**無前綴關係**），**請勿改名為 `/set/sip/multicast_tx`**（`/set/sip/multicast` 是其前綴，else-if 順序放錯即誤命中）；③ handler 收尾 `cJSON_Delete` 與每個錯誤路徑 `free_keyvalue_file` 勿漏。
- **驗收**：無 token → `A003`；`225.1.1.1` 合法、`192.168.1.1` → `E001 非法组播地址`；port 0/65535 → `E001 非法组播端口`；合法請求 → key 寫入 + termapp 於 ~2 秒內啟停推流，全程不重啟。

#### §2.2　GET 擴充（MTX-03 規格）

- `/get/sip/config`：於 `multicast_config` 之後新增（欄位形狀對稱）：
  ```json
  "multicast_tx_config": {"multicast_address": "239.0.0.100", "multicast_port": 9000,
                          "enabled": false, "audio_codec": "G.722"},
  ```
- `/get/device/status`：於 `sip_status.multicast_status` 同層新增：
  ```json
  "multicast_tx_status": {"status": "发送中", "address": "239.0.0.100:9000", "audio_codec": "G.722"},
  ```
  `status` 取值：`发送中`／`关闭`（若採 MTX-08 真實狀態，另加 `故障`）。
- ⚠ **前車之鑑（FW-01/FW-02）**：這兩條端點都是 `snprintf` 手刻 JSON。擴充後務必：① 大括號逐層閉合（`multicast_tx_status` 必須是 `sip_status` 的**直接子物件**、與 `multicast_status` 同層，勿再巢狀錯位）；② `http_msg[2048]` 緩衝確認不截斷；③ **以標準 JSON parser 驗收整包回應**。
- **驗收**：兩端點回應可被 `python3 -m json.tool` 解析；新欄位位置與形狀如上。

#### §2.3　termapp TX 管線（MTX-04 規格）

- 收到 `set_sip_multicast_tx` 指令（或開機時）重讀 `MULTICAST_TX_*`：`ENABLED=true` 即啟動、`false` 即停止。
- 管線：Mic 採集（沿用既有採集，`CAP_VOL` 音量生效）→ 依 `MULTICAST_TX_CODEC` 編碼 → RTP 封包化 → UDP `sendto` 至 `ADDRESS:PORT`。
- 發送 socket：`IP_MULTICAST_TTL=1`（同網段；跨網段見 MTX-09）、**`IP_MULTICAST_LOOP=0`**。
- `ENABLED=true` 時**開機自動恢復推流**（與 RX 行為對稱）。
- **驗收**：另一台 GT-SIP-GW 以既有組播接收設同位址:埠 → **聽到本機 Mic 聲音**；tcpdump 見 RTP 流向組播位址（PT/速率符合 §2.4）；`enabled=false` 後 ~2 秒停播；重開機後自動恢復。

#### §2.4　編碼與 RTP 參數（MTX-05 規格）

> **2026-07-02 決策：v1 編碼固定 `G.722`，放棄 OPUS**。理由：OPUS 之 RTP PT 為動態值（組播無 SDP 協商，須兩端硬寫死）、既有接收端 OPUS 解碼未經驗證、持續編碼 CPU 佔用存疑——三者對 G.722 皆不存在。G.722 為 16 kHz 寬頻語音（廣播喊話音質充分），且為第三方 IP 喇叭/話機組播相容性首選。

| `audio_codec` 值 | RTP PT | RTP clock | 取樣 | ptime | 位元率 | 備註 |
|---|---|---|---|---|---|---|
| **`G.722`（v1 唯一）** | **9（RFC 3551 靜態值）** | **8000**（RFC 3551 特例，實際取樣 16 kHz） | 16 kHz | 20 ms | 64 kbps | timestamp 步進 160/包、50 pps |

接收端 `MULTICAST_CODEC` 同設 `G.722` 即可收聽（**部署規範**：收發兩端一律 G.722，見 MTX-11——已決策不做 PT 自適配）。

- **驗收**：TX 啟用 → 另一台 RX 設 `G.722` 同位址:埠可聽；Wireshark 確認 PT=9、封包 ~50 pps、timestamp 步進 160/包。

### 🟠 P1 — 本版次應做

| 編號 | 要求 | 驗收 |
|---|---|---|
| **MTX-07** | **音訊資源仲裁——✅ 已與原廠確認（2026-07-02）**，三層優先級：**SIP 通話 ＞ Mic 採集（組播發送）＞ 組播播放（接收）**。即：① 發送模式下 SIP 通話進來 → 自動切換至通話；通話結束 → 自動切回 Mic 發送模式；② **Mic 發送啟用期間，本機組播接收不播放**（發送優先於播放）；③ 現行終端既有行為為「SIP 通話＞組播播放」，本功能在其間插入 Mic 採集層 | TX 推流中撥入 SIP 通話 → 推流暫停、通話正常；掛斷 → ~2 秒內恢復推流；TX 啟用期間對本機發組播 → 本機不播放，停用 TX 後恢復播放 |
| **MTX-08** | **真實狀態回報**：termapp 將 TX 實際運行狀態落地 `/tmp/mcast_tx`（照 `/tmp/primary` 既有先例），`multicast_tx_status.status` 優先讀該檔（`发送中/关闭/故障`），無檔再以 config 推導。理由：TX 管線可能失敗（Mic 被占、socket 錯誤），純 config 推導會顯示假「发送中」 | 拔除/占用音訊裝置使 TX 失敗 → 狀態顯示 `故障` 而非 `发送中` |
| **MTX-11**（原廠「格式不一致」問，**✅ 2026-07-03 已決策：不做自適配**） | 韌體**無須**依 RTP PT 自動選解碼器。改以**部署規範**解決：初始設定時，推流端（Mic in）與**所有**組播播放端（line out）之編碼一律設 `G.722`。（本功能 TX 固定 G.722，只要播放端照規範設定即無不一致問題） | 部署/使用文件載明「收發兩端編碼一律 G.722」；混配編碼之解碼錯誤不列為韌體缺陷 |
| **MTX-12**（原廠「多音源」問，**2026-07-03 已回覆方向**）| 多音源處理：**首選混音**（多源同時解碼混音播放）——請原廠評估 CPU/實作可行性；**若不可行，退按 MTX-07 優先級之單源處理**（先到先鎖，以 SSRC＋來源 IP:port 識別，連續 ~2 秒無封包釋放、可接受新源）。SIP 通話仍為最高優先。**請原廠回覆最終採用哪種** | 混音方案：兩源同發 → 兩路皆可聞；單源方案：只播先到者，停止 ~2 秒後自動切另一源 |

## 三、端對端驗收清單（總表）

1. 無 token `POST /set/multicast/tx` → `A003`；非法位址/埠 → `E001`（訊息如 §2.1）。
2. 合法設定＋`enabled=true` → **~2 秒內開播、全程免重啟**。
3. 另一台 GT-SIP-GW 既有組播接收設 `G.722`＋同位址:埠 → **聽到本機 Mic 聲音**。
4. tcpdump/Wireshark：RTP 至組播位址，PT 與封包速率符合 §2.4 表。
5. `enabled=false` → 停播；設備重開機 → `enabled=true` 自動恢復推流。
6. TX 位址:埠 == 本機 RX 位址:埠 → 拒啟（MTX-06）；本機不解播自己的流。
7. SIP 通話期間推流暫停、掛斷恢復（MTX-07）。
8. `/get/sip/config`、`/get/device/status` 整包回應通過標準 JSON parser（MTX-03 ⚠）。
