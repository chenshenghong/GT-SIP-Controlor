# SIP／設備資訊一致性需求單（GT-SIP-GW / 板 v6.1.3）

> 提交對象：設備/韌體原廠
> 背景：實測發現同一台設備、同一項「SIP 伺服器」資訊，在 **DBP 發現協定回應**與 **REST/Web `/get/sip/config`** 兩條路徑回報的值**不一致**；甚至**單一份 DBP 回應內部**就同時存在兩個互相矛盾的「伺服器」欄位。雖不影響現行使用，但會誤導維運人員與上位機（CMS）判斷，且牽動我方自動佈建流程的正確性，本版次請一併修正。
> 依據：對真機 v2.1.1 韌體之 DBP 廣播回應與 REST 回應逐欄比對。
> 提出日期：2026-07-17
> 優先序：🟠 P1 本版次應修（不影響即時使用，但影響維運判讀與上位機自動化）

---

## 一、問題現象（實測）

同一台設備，「SIP 伺服器位址」在下列位置出現**三種來源、兩種值**：

| 來源 | 欄位 | 回報的「伺服器」 |
|---|---|---|
| DBP 回應 · 頂層行 | `Server` / `Server2` | 平台位址 **A**（埠 8800） |
| DBP 回應 · 內嵌 `IFCFG-APP`（base64 JSON） | `RegAddr` : `ServerPort` | SIP 註冊位址 **B**（埠 5060） |
| REST／Web `/get/sip/config` | `primary_line.server_address` : `server_port` | SIP 註冊位址 **B**（埠 5060） |

> 位址以 A／B 匿名表示（避免測試網段細節外流）。關鍵在**值不同**：`Server` 指向平台位址 A:8800，`RegAddr` 與 REST 一致指向 SIP 註冊位址 B:5060。

**維運人員直覺會把 DBP 頂層的 `Server` 當成「SIP 伺服器」**，但 SIP 帳號實際註冊的是 `IFCFG-APP.RegAddr`（＝REST 回報值）。兩者不同 → 產生「DBP 說是 A、Web 說是 B」的誤解。

---

## 二、根因研判

這是**歷史欄位語意衝突**，非即時同步失敗：

- DBP／IPGUARD 舊協定原有 `Server`／`Server2` 一對欄位，歷史語意是「平台／發現／上報伺服器」。
- 但 SIP UA 實際註冊目標存在另一處 `IFCFG-APP.RegAddr` + `ServerPort`。
- 韌體從未把這兩者統一，`Server` 仍指向平台位址、`RegAddr` 才是真正 SIP proxy，兩份資料各存各的、卻都被工具標示為「伺服器」。

換言之：**同一個概念「伺服器」被映射到兩個獨立 config key，且分別餵給不同對外介面**，於是產生矛盾。

---

## 三、需求明細

| 編號 | 現況 | 風險 | 要求 | 驗收 |
|---|---|---|---|---|
| **CON-01** | DBP 頂層 `Server`/`Server2` 與 `IFCFG-APP.RegAddr`/REST 對「SIP 伺服器」回報不同值 | 維運與上位機誤判 SIP 註冊目標 | 所有對外介面（DBP 能力清單、REST、Web）回報的 SIP 配置**必須來自同一份執行期資料源（single source of truth）**。若 `Server`/`Server2` 另有「平台/上報伺服器」語意而非 SIP 註冊位址，**須明確更名區分**（如 `PlatformAddr` vs SIP `RegAddr`），不得共用「Server」一詞造成混淆 | DBP 頂層與 `IFCFG-APP` 及 REST 對「SIP 註冊位址」回報值一致；或欄位名已明確區分兩種語意 |
| **CON-02** | 各介面配置可能為不同時刻快照 | 一介面改配置、他介面回報陳舊值 | 任一路徑修改 SIP 配置後，其他介面回報須在 **≤2 秒內一致、不得要求重啟** | Web 改 SIP 伺服器 → 立即 DBP 查詢，值一致，無須重啟 |
| **CON-03** | 無法區分「配置值」與「執行值」 | 註冊未成功時，回報看不出是「設了但沒註冊上」 | 若需同時表達配置值與執行狀態，用**不同欄位名明確區分**（`sip_config.*` vs `sip_runtime.*`，如 `sip_runtime.registered: true/false`） | 回報可分辨「已設定的伺服器」與「目前是否註冊成功」 |
| **CON-04** | 無配置版本標記 | 上位機無法廉價偵測「設備配置被人動過」或「回報是否陳舊」 | 各介面回報附帶 `config_seq`（配置序號，每次任一路徑修改遞增） | 任一路徑改配置後 `config_seq` +1，且所有介面回報同一序號 |

## 四、附帶發現（同版一併處理，非本單核心）

| 編號 | 現況（DBP 實測） | 要求 |
|---|---|---|
| **BRAND-02（延續）** | DBP 回應 `Type: UNKOWN`——**拼字錯誤**（應為 `UNKNOWN`）且未回報真實設備型號 | 修正拼字；`Type` 回報實際型號代碼，與 REST `device_info.model` 一致（延續《設備 Web 安全強化需求單》BRAND-02） |
| **SEC-10（新增）** | DBP 回應的 `IFCFG-APP` 內 `RegPswd`（SIP 密碼）以**明文** base64 隨廣播回覆——base64 非加密，全網段可嗅 | DBP 回應不應攜帶明文 SIP 密碼；如上位機佈建確需，須遮蔽或改為需鑑權的通道取得（與 REST SEC-02「密碼遮蔽」對齊，避免 REST 已遮蔽、DBP 卻明文外洩的破口） |

> **一致性延伸**：SEC-10 與 REST SEC-02 是同一問題的兩面——若只在 REST 遮蔽密碼、DBP 仍明文廣播，等於沒修。請兩條路徑一併處理。

---

## 五、驗收方法（一句話）

Web 改 SIP 伺服器 → 立即查 DBP 與 REST：三者「SIP 註冊位址」一致、`config_seq` 同步 +1、全程不重啟；且 DBP 回應不再出現明文 SIP 密碼、`Type` 回報正確型號。
