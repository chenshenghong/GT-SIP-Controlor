# 設備 MQTT 能力驗證結果 + 規格索取（給工廠）

> 提交對象：設備/韌體原廠
> 背景：因設備 HTTP web server 丟連線嚴重（見《設備Web服務丟連線診斷報告》），評估改用 **MQTT** 取得狀態 + 大量同步設定。
> 驗證日期：2026-06-22

---

## 一、目的

設備韌體的設定中已有 MQTT 欄位（QueryTool「其他配置 → MQTT设置」可填）。我方希望用 MQTT 取代不穩的 HTTP，達成「即時取得狀態 + 一對多大量下發設定」。本次以實機 + 抓包驗證設備的 MQTT 連線能力。

## 二、驗證環境

| 項目 | 內容 |
|------|------|
| 測試設備 | SipTerm / SIP-Player-2024，IP 192.168.0.147，MAC 20:23:BB:B2:93:E7 |
| MQTT Broker | mosquitto 2.1.2，IP 192.168.0.155 port 1884，**允許匿名**、與設備同網段、TCP 可達（已用 `Test-NetConnection` 驗證 1884 通）|
| 設定方式 | QueryTool「MQTT设置」：勾選「MQTT注册」、设备名称 `siptest147`、Client ID `siptest147`、帳密留空 |

## 三、驗證步驟與結果

1. **寫入 MQTT 設定** → 用 DBP 廣播讀回設備設定確認**寫入成功**：
   - `MQTT_NAME = siptest147`、`CLIENT_ID = siptest147`、`USER_NAME/PASSWD = (空)`
   - `MQTT_URL` 先後測兩種格式皆有正確寫入。
2. **重啟設備套用**（REST `/system/restart`，已確認 uptime 歸零、設備正常開機）。
3. **在 broker 端觀察 + 在 broker 主機 tcpdump 抓 `.147` 封包**。

| MQTT_URL 寫入值 | 重啟 | broker 是否收到連線 | tcpdump `.147`→broker | 結果 |
|---|---|---|---|---|
| `192.168.0.155:1884` | ✅ | 無 | 60 秒 0 個 SYN | ❌ 設備未嘗試連線 |
| `tcp://192.168.0.155:1884` | ✅ | 無 | 60s + 90s 0 個 SYN | ❌ 設備未嘗試連線 |

> 關鍵：設備對 broker（任一格式、重啟後）**完全沒有發起任何 TCP 連線**（連 SYN 都沒有），代表設備並非「連了被拒」，而是**根本沒嘗試連線**。

## 四、結論

- 設備**有** MQTT 設定欄位、且設定**能寫入並持久化**（DBP 可讀回）。
- 但設備配置 MQTT broker、重啟後，**不會主動連線到標準自架 broker（mosquitto，匿名）**。
- 我方僅持有 REST web 設定層源碼（`websetsip.c`），**MQTT client 實作（`termapp`）屬原廠**，無法自行定位連線邏輯與所需格式。

## 五、請原廠協助提供（MQTT 規格）

1. **`服务器地址` / `MQTT_URL` 的正確格式**為何？（我方試過 `host:port` 與 `tcp://host:port` 皆無連線嘗試 —— 是否需要 `mqtt://`、需分開的 port 欄位、或其他寫法？）
2. **除了勾選「MQTT注册」外，是否還有其他啟用步驟 / 旗標**才會讓設備開始連線？
3. **設備 MQTT 是否支援連線到「自架的標準 MQTT broker」**（如 mosquitto / EMQX），還是**只能連原廠指定的雲端平台**？若限定平台，平台的 broker 位址 / 協定為何？
4. **是否需要 TLS（8883）或特定帳密**才會連線？（設定有 用户名称 / 用户密码 欄位）
5. **MQTT 協定規格**：
   - 設備會 **publish** 哪些 topic？payload 格式（JSON 結構）？發佈頻率？
   - 設備會 **subscribe** 哪些 topic（指令通道）？
   - QoS、keepalive、LWT 設定為何？
6. **【最關鍵】設備能否經由 MQTT「下發設定指令」**（如設定音量、SIP 帳號、組播），還是 MQTT **只能上報遙測**？
   - 若能下發 → 我方可用 MQTT 同時做「即時狀態 + 大量同步」，徹底繞開 HTTP 丟連線問題。
   - 若只能上報 → 至少狀態走 MQTT，設定仍需修好 HTTP（見另一份 web 服務診斷報告）。

## 六、與另一份報告的關係

本驗證與《設備Web服務丟連線診斷報告》互補：
- HTTP 路線：需原廠修 `lgw_web` 單執行緒丟連線缺陷。
- MQTT 路線：需原廠提供上述規格，確認可連自架 broker 並支援指令下發。

**兩條可靠性路線都需原廠配合**；請原廠評估哪一條較快可行。
